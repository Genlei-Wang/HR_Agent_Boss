## 需求场景

HR，每天都要在boss招聘网页-候选人列表页面，挨个点开列表里的候选人，判断他的工作经历、项目经历，是否和jd（岗位要求）相关。
- 如果相关，就点击打招呼按钮，然后关闭；
- 如果不相关，就关闭，继续下一个。

* 注意：
    - 候选人详情页-工作经历的网页区域是canvas结构（可以先截图，调用AI大模型，判断后给出结果：符合、不符合）

## 交互设计（草稿）
一个非常简洁、易用的chrome网页插件；
用户安装到chrome，点击后从网页侧边栏弹出；
支持用户输入循环候选人数量（默认99，提醒不要超过100）、职位描述JD；
然后点击按钮「开始打招呼」，就自动运行；按钮变为「停止」，点击后则运行停止；
下面是运行日志（可导出txt），记录了详细信息，如候选人信息（姓名、年龄、公司、学校等），工作经历是否符合，是否打招呼；
整体交互有全面、体验好的提示；

## 其他
在撰写技术方案和开发时，基于“Canvas 截图”需求，有几个核心难点需要注意：

1. 解决 Canvas 数据抓取问题
Boss直聘等平台为了防爬虫，确实会将核心信息（如姓名、电话、经历）渲染在 Canvas 上，导致无法通过 DOM (document.querySelector) 直接获取文本。
技术解法：使用 html2canvas 或浏览器原生 captureVisibleTab 接口对特定区域截图 -> 转换为 Base64 编码 -> 发送给 Gemini 3 Flash（通过 API）。
针对 Canvas 截图判断，Prompts（提示词）要这样写效果最好：
“图中是一个候选人的工作经历截图。我的招聘需求是：[JD详情]。请判断此人是否具备[核心能力]。仅输出 JSON：{"match": true/false, "reason": "理由"}。”

2. 浏览器插件架构 (Manifest V3)
需要一个 Side Panel (侧边栏) UI，这在 Chrome Manifest V3 中体验最好。
需要 Content Script 来操作页面（点击按钮、截图）。
需要 Background Service Worker 来处理 API 请求（避免跨域问题 CORS）。

3. 账号风控 (高危预警) 
这是最需要写进 PRD 的风险点。
问题：如果在短时间内连续点击“查看详情”、“打招呼”，且频率极度规律，Boss 平台的风控系统会识别出这是脚本，进而弹出验证码或直接封禁 HR 账号。

解决方案：
随机延迟：在 PRD 中必须加入“拟人化延迟”，例如每次操作间隔 1-8 秒的随机时间。
每日上限：建议每日运行次数（如你设计的 100 次）
